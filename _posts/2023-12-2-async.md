# 词源

不同的人对术语“同步”、“异步”、“阻塞”和“非阻塞”的定义大有不同，在 Stack Overflow 上，我比较信服的一段关于陈述“异步”这个术语的来源是这么说的：

> 我相信该术语首先用于同步与异步通信。同步意味着两个通信部分有一个共同的时钟信号来运行，因此它们是并行运行的。相反，异步通信有一个就绪信号，因此一部分会请求数据并在信号可用时返回信号。
>
> 然后，这些术语适应了流程，但由于存在明显的差异，术语的某些方面的工作方式有所不同。对于单线程进程，请求执行某些操作的自然方法是进行同步调用，将控制权转移到子进程，然后在完成时返回控制权，然后进程继续。
>
> 异步调用的工作原理与异步通信类似，即您发送要完成某件事的请求，并且执行该操作的进程在完成时返回一个信号。术语用法的区别在于，对于进程来说，异步处理是进程并行运行，而对于通信来说，是并行运行的同步通信。
>
> 因此，“计算机或电机”的范围确实太宽，无法正确定义该术语，因为对于不同的技术，它的使用方式略有不同。

这段回答也直接说明了现在计算机领域中的“异步”概念是从信号领域来的，也说明了“同步”和“异步”这个概念受观察主体的变化而变化。

现实世界中，术语”同步“和”异步“往往是和通信方式相关，例如，打电话是一种同步的通信方式，而在聊天软件、评论区发消息或者用发邮件就是异步的，但是，涉及到计算机领域，当我们将注意力或者观察主体放在一个操作或者一段控制流逻辑上的时候，情况就完全变了。

# 字面含义

先不考虑上下文，只是简单地讨论字面含义；同步的字面意思是”同时发生“，即若干个彼此独立的事物在同一个时间点上做相同的动作，例如小学和中学的课间操，学生在口号或者音乐的节奏中做出整齐的动作，显然他们的动作是同步的。

反之，那些”不同时发生“的是”异步“的，在上述的那个例子中，如果有一个学生乱了动作，就会让在做操的整个方阵变成异步的——即使只有一个学生不同步。

显然，这个概念和信号与通信领域中的同异步定义更接近（其实应该说信号与通信领域中的同异步定义与现实生活中的定义接近）

# 用法的区别

但是在计算机领域中，”同步“这个概念表达的是”顺序的“，表达是多个操作有次序上的要求，只有一个完成了下一个才能开始，需要注意到的是这个定义的内在要求和“同步”的字面意义是相反的：如果对于多个操作来说只有一个完成了才能开始下一个，那怎么才能让这两个事件同时发生呢？这是不可能的；

反之，计算机领域中的“异步”表达的是“没有顺序限制”，显然，只有在多个操作没有顺序的约束时，它们才能同时发生。

也就是说，计算机领域中的异同步概念体现出一种诡异的逆置：计算机领域中的同步概念和异步概念似乎和现实世界中的异同步概念是完全相反的。

# “同时”的含义

其实问题还不止于此，在计算机领域中和现实世界体现得不太一样的概念不止异步和同步，“同时”这个词也是如此。

在现实生活中，“同时”的例子是容易举的，实际上，对于不同的个体来说，我们总是在和别人同时做着不同的事：在我写下这些文字的同时，世界上的其他事情也在同时发生着——绝对不会因为我在做一些事情就停滞了。但是对于计算机来说呢？

可能有的人会说：现代的操作系统和编程语言都是支持以线程作为调度单位的，操作系统的书上也写了：“线程是操作系统进行调度的最小单位”，如果我发起一个线程，并将没有严格约束要求的代码分开，让主线程和新创建的线程各执行一部分，那这两部分代码不就在同时运行了吗？，概念上是这样的，线程确实是并发的手段，也确实可以让没有顺序约束的代码各自执行而不管另一部分，但是两个线程上的代码真的是“同时”执行的吗？

显然这取决于硬件，若我们的硬件拥有多个核，操作系统是可以让这两个线程做到真正的同时运行的（即并行），但是如果我们的硬件只有一个核心，操作系统就只能通过快速上下文切换来制造一种“多个线程在同时执行代码“的假象，这是操作系统的能力，对于我们的代码来说是透明的、非感知的，我们在代码中根本无从判断到底什么时候自己（指代码）是否被暂停过；就好像有些电影中的”时间暂停“超能力一样，除了拥有这个超能力的人，其他的人根本察觉不到时间被暂停过；在计算机中，操作系统无疑是拥有这超能力的，线程（或者是依附于线程的代码）对操作系统的调度根本无从察觉。

所以在不同的抽象层上，或者从不同的角度来看，”同时“这个概念是有区别的，从计算机外部、CPU 或者操作系统的角度看，代码能否同时执行高度依赖 CPU 是否提供了并行计算能力；而对代码来说，只要开启了多线程，就有部分代码在同时运行着（因为从代码的角度看它观察不到调度行为）；对于用户体验来说两种方式都差不多，因为计算机地速度和人地感知不在一个尺度上，人真的很难察觉到线程调度的发生。

那么回顾第一二段：

> 同步的字面意思是”同时发生“

> 只有在多个操作没有顺序的约束时，它们才能同时发生

这些“同时”的意思真的相同吗？

# 逻辑和逻辑执行的实际承担者

看待计算机中的“同时”概念时的不同观点可能来源于在视角的区别，对软件开发者来说，他们总是本能地从代码出发观察计算机世界，但是代码是不会自己执行的，代码只表达逻辑，它不会自己执行，线程才是逻辑执行的实际承担者（当然最终是由 CPU 核心承担的，只不过线程无疑是这一概念在纯软件中的抽象）。而现实世界不太一样，每个人既拥有自己的行为逻辑，同时也是逻辑的承担者，我们总是本能地认为在现实世界中并发和并行都是常有地事情，大量的串行操作反而不正常。从动作的承担者（人）出发，我们很容易看到人不会停滞地等待一个事件的结束，相反，人会在等待事件结束时做别的事情，等待事件结束时别人来通知自己，人可以在等待其他人完成事件地同时做自己地事情，这显然是个异步的过程。

在现实生活中，我们倾向于从逻辑承担者的角度看待事情的同步性，即“两个可以执行动作的实体是否同时执行了一样的动作”，而在计算机中，我们通常从一段代码本身出发看待，即“若两个执行者执行了相同的逻辑，他们在这逻辑上构成严格先后关系吗？”；

我总是觉得从这个角度出发可以理解同异步定义的内在逻辑，但是我的描述可能不是非常严谨，现在它只是一个模糊的影子，还需要更多的实践才能让它变得更明晰严谨。

# 参考系

代码中的情况有所不同，因为我们可以在一段代码中发起多个线程来将一段同步的代码转换为异步的，但问题也随之而来：所谓的异步究竟是对谁而言的，对于异步 IO 来说，我们只是说这个发起 IO 操作的调用（一个 read 函数或 write 函数）是异步的，是可以和我们的其他代码并行的执行的，不会有约束——至少在我们的代码要使用 IO 操作的结果之前是这样的，全程不需要我们做任何同步的操作，在我们要使用 IO 的结果时，也仅仅是调用一个可能同步阻塞的函数来等待结果，这个函数是可能阻塞的事实其实不影响 IO 操作本身是异步的。IOCP 就是这样的，Asio 就基于它实现了 Proactor 模式。

可能有人觉得将一个同步阻塞的 IO 操作放到另一个线程去执行，让另一个线程去等，等到这个同步阻塞IO调用返回了再用一些跨线程通知手段通知主线程同步，不就让 IO 操作变成异步的了吗？但实际上 IO 操作本身依然是同步阻塞的，只是不阻塞主线程罢了，所谓异步非阻塞是说这个IO调用的性质，而不是控制流的性质，开启新线程让原先同步的指令流变成了两个异步的指令流，它们互不约束、并发执行，但没有改变发起 IO 操作的那个调用的性质；

所以一个异步的 IO 操作至少是不强制要求我们发起用户线程来等待 IO 的，实操中可能存在着由操作系统或者异步 IO 库维护的等待 IO 的线程，但这不是我们用户考虑的事情。

# 异步IO和异步操作

涉及到“异步”这个概念的具体场景，最常见的应该是异步IO和异步操作，这两个概念中，IO应该是概念相对小的那一个（毕竟IO操作是一个操作，但不是所有的操作都是IO操作），所以这段文字从IO开始。

## IO的定义

我记得我第一次见到一些高级语言（如C++和Java）中的各种流类时，我总是很难区分那些带有Input和Output的流类的数据流向是从哪里到哪里，后来直到我知道IO的个术语的含义，这个问题才得到解决。IO是一个对内存中的数据的读写过程，所谓Input是将数据从其他设备读入内存，Output是将数据从内存写出到其他设备；可能从不同的方向看，读写关系可能不同，但是数据从其他设备流向内存的一定是输入，而数据从内存流向其他设备的是输出。

## 同步IO和异步IO

要进行IO，实际上是要进行IO操作，从我们代码的角度来看，所谓IO操作就是涉及IO的调用之和，只有执行这些操作之后真正拿到了数据，才说IO操作完成了；但是“拿到数据”又是什么意思？例如在Windows上，在发起IO调用时，调用线程会阻塞直到IO操作完成，一旦线程解除了阻塞且再次获得CPU时间，IO调用就立即返回，这时要么发生了一个错误，可以通过返回值或线程局部错误码检查出来，要么读取成功，读取到的数据已经被写在了调用时提供的地址上；这之中其实包含两个过程：DMA设备将数据写到内核缓冲区、内核将数据从内核缓冲区写到调用者指定的地址。

这样的过程是同步的，也是阻塞的，说阻塞是因为进行IO调用后到返回之前，调用线程是阻塞着的，这时候线程根本无法被调度（因为IO还未完成），说同步是因为这IO操作本身和同一指令流中的其他指令无法同时（严谨的说是某一层面上的并发）执行，即使IO操作根本不会占用CPU（如果它占用CPU的话调用线程就不会被阻塞了）。这种方式十分简单无脑——我在等一件事发生，如果他没发生我（线程）就一直躺着不动，只能说非常巴适；而且从指令流的角度来看也比较简单明了——利用语言定义的作用次序将依赖IO操作结果的指令严格的限制在IO操作之后，写出来的代码是线性的，依赖关系一看就知。

但是并非所有的代码的要求都如此的线性，服务器标准模型所面临的问题就是如此：服务器的逻辑运行在一个循环中，不断的接受连接、处理请求并发回响应数据，次序上看，循环中的逻辑既先于IO操作又后于IO操作，这就意味着如果IO操作是阻塞同步的，不仅每次进入阻塞状态和阻塞恢复需要额外的时间成本（来自操作系统的调度），同步也导致了IO操作未完成时逻辑上不依赖于它的接受连接操作也无法执行。若操作系统只提供阻塞同步的IO操作，那应用就不得不承担这些成本——要么只能串行地处理连接，要么使用多个线程、承担内核级线程的使用成本。

所以在实际开发服务器应用时需要一种异步非阻塞的IO方法：调用发起IO的函数之后线程不阻塞，立即返回，这个线程可以立即做其他不依赖IO结果的事情，而依赖IO结果的逻辑以其他方式处理（例如：在IO完成的线程上回调、在IO完成后被加入到发起IO的线程的事件循环中稍后在这个线程上执行）

## 阻塞行为

和混乱的异步相比，阻塞的定义就太清晰了：阻塞是操作系统的概念，完全基于线程的概念定义；阻塞是线程由于等待一个事件发生而进入的一个状态，进入阻塞状态的线程不能获得CPU时间，直到它等待的事件到达完成，操作系统会将这线程放入就绪队列，线程会在到达就绪队列头后重新获得CPU时间。

可见一个线程被阻塞，往往是因为它要等待的事件是由其他线程或其他设备执行的，它只要等着就好，即使获得了CPU时间也没有用（因为按照编程语言规定的内存顺序，后于它的表达式和它有着次序上的依赖，不能在它还未完成之前求值），这是操作系统为了节约CPU时间的必然选择，毕竟不能让一个啥都不能干的线程白占着CPU。

## 分类

上面说过，在操作系统上调用读写系统调用进行IO操作，实际上包含两个过程：

> 这之中其实包含两个过程：DMA设备将数据写到内核缓冲区、内核将数据从内核缓冲区写到调用者指定的地址。

同步IO阻塞等待，实际上就是等待这两件事情完成，但是若改阻塞IO为非阻塞，那编写代码的程序员就得到了更大的灵活性，如下：

```c
void function() {
    doSomething();
    while(!NonBlockingRead(pointer)) {
        doSomething();
    }
}
```

非阻塞IO系统调用可以放在一个死循环中，这个调用在IO完成之前会直接返回而不是阻塞，所以在循环中还可以“腾出手”来做一些其他工作，而且在操作系统的非阻塞IO实现中，这个调用只会在第二个过程时阻塞的，这实际上缩短了线程的必要阻塞时间；所以非阻塞IO事实上是“半阻塞”的，它并未彻底改善IO的问题。

非阻塞IO应该称为同步非阻塞IO；

改善同步阻塞IO问题的另一个途径是多路复用，和异步一样，这个词也来自信号和通信，就像通信时复用信道那样，IO多路复用复用了线程，使一个线程可以处理多个IO。具体的做法是将IO操作委托给系统，系统会维护IO事件和IO完成事件的队列，系统会另外提供API获得IO完成事件和对应的句柄，这就是所谓的解复用。
